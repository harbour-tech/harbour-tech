Web3 SDK

Harbour does all the heavy lifting, such as performing identity verification on customers, authenticating them, and creating the necessary bank accounts. Integrating apps simply have to deal with basic information retrieval for rendering their UI, and taking care of signatures. See how to perform a basic integration with Harbour in under 5 minutes, using the Harbour SDK.

Our SDK is written in TypeScript, but sources can be generated for any language that supports buf. Buf Connect is a client-server protocol that allows communication over http(s) via either JSON- or binary-encoded protobuf messages.

Unlike a REST API, all calls happen through an http post strictly adhering to a type-safe contract definition.

Our protobuf definitions clearly expose all invokable methods (a.k.a. RPC: remote procedure call) and their expected input and output types and serve as living documentation of the API.

Here's a simple example you can try yourself on the command line with an anonymous endpoint:

curl -X POST -H "Content-Type: application/json" -d '{"fiatAssetId": "CURRENCY_ID_EUR"}' https://api.harborapp.link/ramp.v1.RampService/GetAssets
{
  "cryptoAssets": [
    {
      "status": {},
      "protocol": "PROTOCOL_ETHEREUM",
      "network": "NETWORK_ETHEREUM_MAINNET",
      "assetId": "ASSET_ID_USDC",
      "shortName": "USDC",
      "symbol": "USDC",
      "ethereumErc20Token": {
        "tokenAddress": "0xA0b86991c6218b36c1d19D4a2e9Eb0cE3606eB48"
      },
      "onrampLimits": {
        ...
      },
      "offrampLimits": {
        ...
      }
    },
    ...
  ]
}

The advantage of protobuf is that developers don't need to know the specifics of the protocol, or how messages are serialized, and simply interact with function calls on the generated sources.

API versioning​

Buf messages are backwards and forward compatible as long as no fields are renamed, renumbered or removed. We might introduce slight changes such as adding new fields and new RPCs over time, maintaining compatibility. Eventually, we'll resort to versioning in case of breaking changes and promptly inform our API clients.

Conventions​

Numeric amounts are represented as strings of arbitrary precision. The format is the standard floating-point with the dot as decimal separator. Eg: 1100.24.

Authentication​

Harness the power of a crypto native, password-less, wallet-centric authentication.

Unless specified otherwise, all RPCs require authentication headers, as described in the authentication overview.

Our API is designed to authenticate clients via cryptographic keys, making it suitable for crypto wallets and other decentralized applications. Every API request must contain a set of headers specifying a public key, specifics of the digital signature algorithm use, and a signature of the request payload. Any request with a valid signature is considered "authenticated", for the given public key.

Harbour associates one or more public keys with a customer's identity, with a customer being uniquely identified with their full name, date of birth and residential address, and further authenticated via SMS OTPs.

Once a customer is onboarded, the public key is associated with them, and they can start accessing the ramp functionality. Multiple keys can be associated with the same customer, via a special sign in confirmation procedure.

This allows for a crypto wallet to seamlessly interact with the Harbour API without having to store any sensitive information, nor API keys, nor asking user passwords. All it needs to do is sign each request with a key. By deriving the key deterministically from the seed, the end user will be able to access the Harbour API from any device they install their wallet on, without having to re-authenticate each time.

Anonymous endpoints​

Assets and quotes can be retrieved anonymously. Check out:

  • the GetAssets RPC for a list of supported assets and blockchains
  • the EstimateOnRampFee and EstimateOffRampFee RPCs for estimating the fees and the expected amounts when buying or selling crypto

Note that the processing fee is always in fiat, 50 euro cents in the example below:

curl 'https://snap.harbour.fi/api/ramp.v1.RampService/EstimateOnRampFee' \
  -H 'content-type: application/json' \
  --data-raw '{"fiatAssetId":"CURRENCY_ID_EUR","cryptoAssetId":"ASSET_ID_USDC","protocol":"PROTOCOL_ETHEREUM","fiatAssetAmount":"100"}'
{
  "fiatAssetAmount": "100",
  "cryptoAssetAmount": "101.77",
  "exchangeRate": "1.0438",
  "networkFeeAmount": "2",
  "processingFeeAmount": "0.5"
}

Note that we only support EUR to crypto and vice versa, so the list of assets represents any asset that can be bought with EUR or sold for EUR. As we add support for further fiat currencies, we will provide more details on fiat assets available for customers, depending on their region.

There is no endpoint specifying the payment method, as we only support bank payments, so that's implicit as well.

The fee estimation endpoints quote the current exchange rate between fiat and crypto, a service charge and the expected network fees. The resulting amount of crypto that can be bought with a given fiat amount, or the resulting fiat that can be received for selling a given amount of crypto, is also returned.

Note, however, that these are not binding quotes, but simply estimations based on the current fair market value of the fiat/crypto pair and the current network congestion. The actual amount is calculated at the moment of payment receipt.

Onboarding​

The very first API call that should be done to Harbour is the GetAccountInfo. This is a "smart" RPC that detects whether the authenticated pubkey is already associated with a customer. If so, it returns all information required to render a full UI to the user, in a single response. See below:

curl 'https://snap.harbour.fi/api/ramp.v1.RampService/GetAccountInfo' \
  -H 'content-type: application/json' \
  --data-raw '{}'
{
  "account": {
    "wallets": [...],
    "onrampIban": {
      "iban": "NL39MODR0004495012"
    },
    "offrampIban": {
      "iban": "LT213250070000973519"
    },
    "accountHolder": "Martin Bechtle",
    "cryptoAssets": [...],
    "limits": {
      "onrampSingleLimit": "2000",
      "onrampDailyLimit": "2000",
      "onrampCurrentDailyUsage": "0",
      "offrampSingleLimit": "2000",
      "offrampDailyLimit": "2000",
      "offrampCurrentDailyUsage": "0"
    },
  }
}

Based on this response, the integrating app can show the customer their magic bank account number, the list of supported assets, their limits expressed in EUR (will expand the API to explicitly state the fiat currency once we support more than one), and a list of whitelisted addresses they can ramp from / into.

However, should the pubkey be unknown to Harbour, the response will show an onboarding URL, such as:

{
  "authentication": {
    "authenticationUrl": "https://example.com/authenticate",
    "expires_in_sec": 3600
  }
}

The integrating app should either redirect or embed an iframe with the above URL, where our web app will take care of onboarding the user. Based on their phone number and personal information, Harbour will detect whether it's an existing customer with a new pubkey, or a completely new customer, and navigate them accordingly to either sign in up and going through identity verification, or just signing in. Either way, at the end of the process, the pub key will be associated with the customer, and from that moment onwards the GetAccountInfo RPC will return the full account info.

Note that sign up is a semi-automated process, and sometimes we need to manually verify a customer's identity, so their account might not be ready for a few minutes to a few hours. In that case, this RPC will still return an onboarding URL. Our onboarding web app will detect the customer status and inform them about their progress in the onboarding process.

Whitelisting and ramping​

At present, Harbour customers can only ramp in a closed loop, meaning:

  • they can only send and receive fiat payments from bank accounts in their own name
  • they can only send and receive crypto payments from wallets they own

Any payment sent from a bank account with a different account holder will immediately bounce back. Any payment sent from Harbour to the user will explicitly state the user's full name as the beneficiary. Harbour will only send crypto to whitelisted addresses, and will only recognise user crypto payments from whitelisted addresses.

Note how the GetAccountInfo RPC returns a list of wallets. These are the on-chain addresses whitelisted so far. When pushing a user through a ramp journey, whether that's buying or selling, the client app needs to ensure that the address the user intends to receive crypto on, or send crypto from, is whitelisted, specifically for the given protocol (blockchain).

In our API, a protocol uniquely identifies a blockchain, see the networks page for more information.

Once a user chooses which account in the HDWallet to use, and the network, the client can check whether the (network, address) is already whitelisted, if not invoke the WhitelistAddress RPC. Our code samples provide a clear example on how to do so, but it's quite trivial: the RPC expects a signature field, which is built simply by signing the address (in hex format) with the private key associated with the address public key.

Finally, when selling crypto, the user needs to specify the bank account they want to receive the fiat payment on. Once Harbour receives the crypto, it will send fiat from the user's magic bank account, to the destination (offramp) bank account.

The offramp bank account can be saved with the SetBankAccount RPC, and from that moment onwards will appear in the GetAccountInfo. The user should be able to change it anytime through the client UI. It is important to implement error handling on the RPC response, as we'll verify whether the account coordinates are valid before saving them.

Example of an unsuccessful request/response:

curl 'https://snap.harbour.fi/api/ramp.v1.RampService/SetBankAccount' \
  -H 'content-type: application/json' \
  --data-raw '{"iban":{"iban":"LT2132500700973519"}}'
{
  "errors": [
    "ERROR_BANK_NUMBER_INVALID"
  ]
}

Note that for now we only support EUR, so the bank coordinate is in the form of an IBAN. As we add support for more fiat currencies, such as GBP, the client app will have to implement different forms for different countries.

Signing specification​

Harbour allows client apps to use cryptographic primitives that are native to their ecosystem. When signing a request payload, there's a bunch of different options available:

  • EVM: sign a keccak256 digest of the request body with a secp256k1 signature
  • Cosmos: sign a sha256 digest of the request body with a secp256k1 signature
  • Substrate: sign a blake2 digest of the request body with a sr25519 signature

Supported hashing algorithms:

  • keccak256
  • sha256
  • blake2b256
  • ethereum (like keccak256, but adding prefix \x19Ethereum Signed Message:{length} before hashing)

Supported signing algorithms:

  • secp256k1
  • sr25519

Supported encodings:

  • hex
  • base64

All the client app needs to do is specify the following http headers:

  • X-Encoding: specifies which encoding to use for all headers containing binary data
  • X-Signature-Type: hasing_algo/signing_algo, eg keccak256/secp256k1 for Ethereum
  • X-Signature: the signature
  • X-Signature-PublicKey: the public key
  • X-Signature-Timestamp: a timestamp to be used as nonce

Note: the timestamp has to be the Unix epoch milliseconds, and has to be within 1 minute of the server time. This implies that a client device with a skewed clock won't be able to authenticate successfully.

And, by the way, header management is abstracted away by our SDK, should you decide to use it.

The signature is produced by:

  • converting the timestamp to its string representation and taking the resulting Unicode bytes
  • concatenating the request body bytes with the timestamp Unicode bytes
  • hashing with the specified hashing algorithm
  • signing with the specified public key and signing algorithm

Whitelist signature​

Harbour requires customers to whitelist their on-chain addresses before they can ramp in or out of it. Whitelisting requires producing a signature with the address' private key, to demonstrate ownership of the address. Ownership is a fundamental requirement for customers using our services.

The signature is produced by:

  • Taking the address's on-chain string representation, e.g. 0x1234567890abcdef on Ethereum
  • Signing such payload with the exact same signing function used to authenticate requests, using the private key associated with the address

  • Examples available on request for other chains

Understanding signing keys​

So far we discussed producing signatures for two different use cases:

  • authenticating requests
  • whitelisting addresses

For authenticating request, you'll want to choose a key which is derived from the user's seed phrase. As long as the key is deterministically derived, as user can install their wallet on multiple devices and not have to go through the process of linking a new key to their account each time.

An example of a deterministic derivation would be simply taking the first key in the HDWallet derived from the seed phrase (e.g. m/44'/60'/0'/0/0 for Ethereum).

On the other hand, for whitelisting addresses, you'll want to use the key associated with the address.

Another important aspect is the ability to access the private key used for signing requests transparently for the user, without having to bother them every time with a signing pop up.

5 minute integration challenge​

Let's integrate your crypto wallet with the TypeScript SDK in just 5 minutes with less than 50 lines of code!

Please note that all examples are based on version 6.x of ethers, which might slightly differ from 5.x.

We'll try producing a valid signature and invoking our GetAccountInfo API with an Ethereum key. Let's use hex encoding and keccak256/secp256k1 for the signature.

Step 1: implement the signing function as per spec. Here's a simple example using an ethers wallet:

import {keccak256, toUtf8Bytes, Wallet} from 'ethers';
import crypto from 'crypto';
import {EthereumSignature, RampClient, Signature} from "../index";
import {GetAccountInfoRequest} from "../gen/ramp/v1/public_pb";

// generate a random wallet, or use your own HDWallet
const privateKey = "0x" + crypto.randomBytes(32).toString('hex');
const wallet = new Wallet(privateKey);

// Our SDK already constructs the required payload by concatenating the request body with the current timestamp.
// Eg: {"foo":"bar"}1234567890.
// You just need to implement a simple signing function such as this one.
const signPayload = (payload: string): string => {
  const hashed = keccak256(toUtf8Bytes(payload));
  return wallet.signingKey.sign(hashed).serialized;
}

Step 2: instantiate the RampClient from our SDK

const ramp = new RampClient(
  "https://dev-api.harborapps-nonprod.link",
  (payload): Promise<Signature> => {
    const sig = signPayload(payload);
    return Promise.resolve({
      signature: sig,
      publicKey: wallet.signingKey.publicKey,
      ...EthereumSignature, // this specifies the correct hashing and signing headers
    });
  },
);

Step 3: get account info

const accountInfo = await ramp.getAccountInfo(new GetAccountInfoRequest())
console.log("Account Info: ");
console.log(accountInfo);

Ethereum personal message signing​

Note that Ethereum wallets typically support raw signing and personal message signing. Depending on which operation you decide to use, you might need to tweak the RampClient instantiation. To be clear, raw message signing is simply signing the keccak hash, whereas personal message signing is signing the keccak hash of the message prefixed with "\x19Ethereum Signed Message:\n{msg_length}".

Following is an example working with personal message signing:

import {Wallet} from 'ethers';
import crypto from 'crypto';
import {EthPersonalSignature, RampClient, Signature} from "../index";
import {GetAccountInfoRequest} from "../gen/ramp/v1/public_pb";

// generate a random wallet, or use your own HDWallet
const privateKey = "0x" + crypto.randomBytes(32).toString('hex');
const wallet = new Wallet(privateKey);

// Our SDK already constructs the required payload by concatenating the request body with the current timestamp.
// Eg: {"foo":"bar"}1234567890.
// You just need to implement a simple signing function such as this one.
const signPayload = (payload: string): string => {
  return wallet.signMessageSync(payload);
}

const ramp = new RampClient(
  "https://dev-api.harborapps-nonprod.link",
  (payload): Promise<Signature> => {
    const sig = signPayload(payload);
    return Promise.resolve({
      signature: sig,
      publicKey: wallet.signingKey.publicKey,
      ...EthPersonalSignature, // this specifies the correct hashing and signing headers for personal msg signing
    });
  },
);

const accountInfo = await ramp.getAccountInfo(new GetAccountInfoRequest())
console.log("Account Info: ");
console.log(accountInfo);

Test user​

Once you manage to get a pub key working against our dev API, it's probably best to store the private key somewhere to facilitate integration work. You can create a test user by following the onboarding URL and select Germany as a country. Insert number: +49 7777908100 and confirm with OTP 908100. Finally, answer the security questions with 90147 as post code and 01/01/1989 as date of birth.

Your pub key is now associated with a test user, and you can interact with all the RPCs to get account info, set the offramp bank account, whitelist addresses, etc.
