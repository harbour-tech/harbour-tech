Web3 Authentication

A special feature of Harbour’s APIs for Crypto Wallets and dApps is the ability to recognise digital signatures from a customer’s wallet. For each wallet a Customer interacts from, we map the public key to the customer in our database. This way, a returning Customer will never have to go through a login step or remember a password.

The only instance in which a login is still required is when a Customer comes from a new wallet (new seed), in which case they’ll briefly confirm their identity by an SMS OTP and a security question. Similar to the Onboarding Process, the authentication process for a new wallet is handled entirely by Harbour and requires no integration work except for iframe embedding or web redirect.

In the Harbour API, there's two ways a user can authenticate themselves through their public key. Either by having the client app sign every message, suitable for apps that have immediate access to the key and signing functionality, or by having the client app sign a special authentication message, suitable for other use cases such as dApps, which will prompt the user's wallet to sign the message and then obtain a session token.

More details below.

Message Signing​

This method is most suited for crypto wallets that would like to embed Harbour in their own UI. They can simply derive a key deterministically from the HD wallet and sign each message with it. Harbour supports several different digital signature algorithms, so that each wallet can use cryptographic primitives that are native to their ecosystem, such as signing a keccak256 digest with a secp256k1 key on EVM chains.

In this configuration, every API request will contain special headers specifying the signature type and the signature of the request body.

With this method, each API interaction will be considered an "authenticated wallet" from Harbour's point of view, which might or might not yet be associated with an "authenticated customer".

Session Token​

This is better suited for web or mobile applications that do not have control over the user's keys. In this case, the web3 SDKs can be used to prompt a user's wallet to sign a custom authentication message, such as 'harbour-auth:timestamp'. The signed message is then sent to Harbour's API, which verifies the signature and returns a session token. Any subsequent API interaction can include this token, which has a limited lifetime.

The session token will be considered an "authenticated wallet" from Harbour's point of view, the exact same as with the above-mentioned message signing method.
